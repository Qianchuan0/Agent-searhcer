## A. 数据结构与落库

### A1. 主承接池条目会持久化 `core_claim`

- 验收目标：
  - 确认 `research_knowledge` 和 `report_index` 已正式保存 `core_claim`。
- 操作步骤：
  1. 完成一次新的研究报告。
  2. 打开 `data/memory/memory.json`。
  3. 找到这次新生成的 `report_index`。
  4. 至少保存 1 条 `research_knowledge` 建议。
- 预期界面：
  - 报告正常完成。
  - 记忆建议流程仍能正常使用。
- 预期数据：
  - `report_index` 至少包含：
    - `title`
    - `core_claim`
    - `summary`
    - `content`
  - 保存后的 `research_knowledge` 条目中包含 `core_claim`。

### A2. 报告会持久化承接元数据

- 验收目标：
  - 确认报告会保存“本次选中了哪些历史记忆”的 ID 和快照。
- 操作步骤：
  1. 发起一个会触发承接弹层的相关研究问题。
  2. 在弹层中只选择 2 条历史结论。
  3. 继续研究并等待报告完成。
  4. 打开 `data/reports.json`。
  5. 找到最新报告对应的数据。
- 预期界面：
  - 报告正常完成。
- 预期数据：
  - 存在 `adopted_memory_ids`，且长度为 2。
  - 存在 `adopted_memories_snapshot`，且长度为 2。
  - 每条 snapshot 至少包含：
    - `id`
    - `title`
    - `core_claim`
    - `summary`
    - `reportId`
    - `score`
    - `confidence`
    - `createdAt`
    - `staleness`

### A3. 老数据缺少 `core_claim` 时不会读挂

- 验收目标：
  - 确认对旧版记忆数据仍然兼容。
- 操作步骤：
  1. 在 `memory.json` 中手动准备一条没有 `core_claim` 的 `research_knowledge` 或 `report_index`。
  2. 重启后端。
  3. 打开应用并触发一个相关研究问题。
- 预期界面：
  - 页面能正常加载。
  - 承接弹层仍可正常打开。
- 预期行为：
  - 这类旧条目不会被当成完整结构化主承接结果。
  - 它可以作为较低优先级的 fallback 候选出现。

## B. 报告完成后的结构化沉淀

### B1. 报告保存后会生成或更新 `report_index`

- 验收目标：
  - 确认每份报告都会被沉淀为可复用的报告索引。
- 操作步骤：
  1. 发起一次不依赖旧历史的新研究。
  2. 等待报告保存完成。
  3. 打开 `data/memory/memory.json`。
- 预期数据：
  - 存在该报告对应的 `report_index` 条目。
  - `source.report_id` 与该报告 ID 一致。
  - 条目中存在 `core_claim`。

### B2. 建议候选中已经带有 `core_claim`

- 验收目标：
  - 确认 `research_knowledge` 建议在保存前就已经结构化。
- 操作步骤：
  1. 完成一次研究报告。
  2. 等待记忆建议面板出现。
  3. 检查建议卡片默认内容。
  4. 如有 `research_knowledge` 建议，再进入“编辑后保存”状态检查可编辑字段。
- 预期行为：
  - `research_knowledge` 建议卡片中能直接看到 `core_claim` 候选，不需要先打开网络面板。
  - 进入编辑态后，`core_claim` 会以独立字段出现，而不是和正文混在一起。
  - `research_knowledge` 建议数据中带有 `core_claim`。
  - 保存后该 `core_claim` 会被保留下来。

## C. 检索与关系分类

### C1. 主承接池优先于辅助背景返回

- 验收目标：
  - 确认检索优先返回结构化历史结论。
- 操作步骤：
  1. 保证记忆库中至少有：
     - 1 条 `research_knowledge`
     - 1 条 `report_index`
     - 1 条 `saved_context`
  2. 输入一个与三者都有关的问题。
  3. 打开承接弹层。
- 预期界面：
  - 主区域先看到结构化历史结论。
- 预期行为：
  - `research_knowledge` 排在 `saved_context` 前面。
  - 当相关度相近时，`report_index` 也应排在纯辅助背景前面。

### C2. `user_preference` 不进入承接链路

- 验收目标：
  - 确认非研究结论类记忆不会混入承接弹层。
- 操作步骤：
  1. 新增至少 1 条 `user_preference` 记忆。
  2. 输入一个会与这条偏好词面命中的问题。
  3. 触发承接分类 / 承接弹层。
- 预期行为：
  - `user_preference` 不会出现在承接弹层中。

### C3. 关系分类会返回稳定的策略说明

- 验收目标：
  - 确认后端返回的分类信息可直接供前端消费。
- 操作步骤：
  1. 通过 UI 或直接调用 `/api/memory/classify-research`，分别测试：
     - follow-up：`继续研究 AI IDE 市场格局`
     - refresh：`刷新一下 AI IDE 市场最近变化`
     - compare：`对比 Cursor 和 Windsurf`
  2. 检查返回的 JSON。
- 预期数据：
  - 返回中包含：
    - `relation`
    - `reason`
    - `suggested_strategy`
    - `related_memories`

## D. 承接弹层

### D1. 主列表默认显示 `core_claim`，不是长正文

- 验收目标：
  - 确认弹层优先展示“可承接结论”而不是大段内容。
- 操作步骤：
  1. 在已有结构化记忆的前提下触发承接弹层。
  2. 检查主列表内容。
- 预期界面：
  - 每张主卡片默认显示：
    - 勾选框
    - `title`
    - `core_claim`
    - 来源报告
    - 创建时间
    - 可信度
    - 时效性
    - 相关度
  - 默认态不展示长正文。

### D2. 辅助背景会进入单独折叠区

- 验收目标：
  - 确认 `saved_context` 和 `research_interest` 不与主承接结论混排。
- 操作步骤：
  1. 保证存在辅助背景类记忆。
  2. 触发承接弹层。
  3. 展开辅助背景区。
- 预期界面：
  - 主列表中只显示主承接池条目。
  - 辅助条目只出现在辅助区。
  - 辅助条目默认不勾选。

### D3. fallback 候选会被明确标记

- 验收目标：
  - 确认缺失 `core_claim` 的旧主池数据被降级处理且界面可辨识。
- 操作步骤：
  1. 准备一条没有 `core_claim` 的主池记忆。
  2. 触发承接弹层。
- 预期界面：
  - 该条目会以 fallback / 临时候选形式出现。
  - 它与正式结构化条目在视觉上有明显区分。

### D4. 默认勾选偏向高质量 `research_knowledge`

- 验收目标：
  - 确认默认选择策略不是简单前 N 条。
- 操作步骤：
  1. 准备多条相关度和可信度不同的承接候选。
  2. 触发承接弹层。
- 预期行为：
  - 高相关、高可信的 `research_knowledge` 会优先被默认勾选。
  - 辅助背景不会被自动勾选。

## E. 新研究注入

### E1. 注入内容改为结构化历史上下文

- 验收目标：
  - 确认研究注入不再是旧报告正文拼接。
- 操作步骤：
  1. 在承接弹层中选择 1 条或多条历史结论。
  2. 点击继续研究。
  3. 通过日志或请求链路检查实际注入的 prompt 文本。
- 预期内容：
  - 每条被承接的历史结论至少包含：
    - `title`
    - `core_claim`
    - `summary`
    - 来源报告
    - 创建时间
    - 可信度
    - 时效性
- 预期行为：
  - 注入内容是结构化的。
  - 整篇旧报告不会作为主上下文被直接拼接进去。

### E2. 不同 relation 会附带不同写作约束

- 验收目标：
  - 确认 `follow_up`、`refresh`、`compare` 注入策略不同。
- 操作步骤：
  1. 分别构造 3 类问题：
     - follow-up
     - refresh
     - compare
  2. 检查各自注入的 prompt 文本。
- 预期内容：
  - `follow_up`：强调新增补充。
  - `refresh`：强调复核旧结论是否仍成立。
  - `compare`：强调新旧差异。

### E3. 使用承接时一定会附带硬约束

- 验收目标：
  - 确认研究任务被明确要求区分历史结论和本次新增发现。
- 操作步骤：
  1. 选择至少 1 条历史结论。
  2. 继续研究。
  3. 检查实际注入的 prompt 文本。
- 预期内容：
  - 历史结论被视为“带来源的旧研究判断”。
  - 历史结论不能直接当作本次已验证事实。
  - 报告必须区分：
    - 历史研究引用
    - 本次新增发现

    - 来源报告
    - 创建日期
    - 可信度

### F2. 报告顶部单独展示“本次新增发现”

- 验收目标：
  - 确认本次新结论与历史引用分开展示。

  1. 完成一次带承接的研究流程。
  2. 打开最终报告页。

  - 出现单独的 `本次新增发现` 区块。

### F3. 报告只回显用户选中的历史结论

- 验收目标：

- 操作步骤：

  2. 只选其中 1 条或 2 条。
  3. 完成研究。
  4. 打开报告页。
- 预期行为：
  - `历史研究引用` 中只出现被选中的条目。

## G. 兼容与安全

### G1. 关闭长期记忆后，不读取也不写入承接链路

- 验收目标：
  - 确认长期记忆关闭时，主研究流程仍然安全可用。
- 操作步骤：
  1. 在 UI 中关闭长期记忆。
  2. 发起并完成一次普通研究。
  3. 再发起一个相关追问问题。
- 预期行为：
  - 不会出现承接弹层。
  - 不会新增长期记忆索引。
  - 普通历史、报告、追问流程仍然正常。

### G2. 含敏感信息的报告不会生成长期记忆建议

- 验收目标：
  - 确认敏感内容不会进入长期记忆建议链路。
- 操作步骤：
  1. 构造或模拟一份包含 `api_key=...` 等敏感内容的报告。
  2. 完成报告流程。
- 预期行为：
  - 不会生成长期记忆建议列表。
  - 首页研究页和历史研究页都不会出现建议卡片。
  - 用户能收到明确阻断提示；除了 toast 之外，页面内也会出现“本次不生成长期记忆建议”的可见说明。

## H. 损坏恢复

### H1. `memory.json` 损坏时不影响研究启动

- 验收目标：
  - 确认记忆子系统损坏时，应用不会整体崩掉。
- 操作步骤：
  1. 停掉后端。
  2. 把 `data/memory/memory.json` 改成非法 JSON。
  3. 重启后端。
  4. 打开应用并发起一个研究问题。
- 预期行为：
  - 研究仍然可以启动。
  - 记忆 UI 会退回到空 / 默认状态。
  - 页面不会因为记忆文件损坏而崩溃。
  - 原损坏文件会被转存为同目录下的 `memory.json.corrupt-*` 备份文件，便于排查和手工恢复。

### H2. 当前长期记忆子系统无独立 `embeddings.json` 依赖

- 验收目标：
  - 明确当前本地优先 MVP 中，长期记忆检索不依赖单独的 `embeddings.json` 文件。
- 操作步骤：
  1. 检查 `backend/server/memory_store.py` 与长期记忆读写链路。
  2. 确认长期记忆只读取 `data/memory/memory.json`。
- 预期行为：
  - 损坏恢复链路当前针对 `memory.json` 生效。
  - 不存在必须额外恢复的长期记忆 `embeddings.json` 文件依赖。
